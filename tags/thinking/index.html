单例模式(Singleton)

原话是：

Ensure a class has only one instance,and provide a global point of access to it.

确保一个类只有一个实例，并且向整个系统提供这个实例。

java中实现单例模式通常有两种表现形式：

  • 饿汉模式单例：类加载时，就进行对象的实例化。
  • 懒汉模式单例：第一次引用类时，才进行对象的实例化。

饿汉模式

知识点：

static关键字：static修饰的变量或方法是属于类的，变量在类加载的时候就已经实例化了。static修饰的方法所有的对象使用的是同一个方法。

private关键字：这里Singleton只能生成单个的实例的根本原因是构造方法被private修饰，不能通过对象调用。只能通过static修饰的静态方法调用。

懒汉模式

知识点：

synhronized关键字：同步，确保在多线程的环境下也是单例，因为，当多个线程同时调用这个方法时，由于使用了synhronized修饰，所以产生锁。允许最先到的线程访问，其他线程等待。当锁消除时，再去访问，实例已经被创建了。直接返回即可。

为什么懒汉模式不用sunhronized修饰呢？因为实例已经被创建了。即使再多线程中同时调用也不会产生多例。

单例模式的优点

  • 频繁被创建和销毁的场景里，使用单例模式减少内存开支。
  • 创建一个对象需要较多的资源和性能开支的场景里（读取配置文件、产生其他依赖对象时），使用单例模式减少内存开支。
  • 避免多重占用（如：写文件）。避免了同一资源的同时操作。
  • 可以在系统设置全局访问点，优化和共享资源。（如：设计一个单例类，负责所有数据表的映射处理）

单例模式的缺点

  • 无法创建子类，扩展困难。（扩展只能修改代码）
  • 不利于测试。在并行开发的环境中，若采用单例模式的类没有完成，程序无法测试。
  • 由于单例类通常不实现接口。所以无法使用mock的方式虚拟一个对象。(我也不知道mock是什么方式)。

使用场景

  • 要求生成唯一序列号的环境。
  • 项目需要一个共享访问点或共享数据。（如：web页面中的计数器，不必存放到数据库中使用单例模式能保持数值，并能确保线程安全）
  • 创建一个对象需要消耗的资源过多，如访问IO和数据库等资源。
  • 需要大量静态常量和静态方法（如：工具类）的环境，（当然，也可以直接采用static声明的方式）。

在Spring框架中每个Bean默认的就是单例模式。java基础库中java.lang.Runtime也是单例模式，其中getRuntime()方法返回了唯一实例。

注意事项

单例类分为有状态和无状态:

  • 有状态单例类：有状态的单例类一般是可变的，通常当作状态库使用。如：给系统提供唯一的序列号。
  • 无状态单例类:无状态单例类的对象一般是不变的，通常用来提供工具性的功能方法。如：IO或数据库等。

设计一个模式的过程就是将问题抽象化，忽略不重要的细节后发现问题本质，并找到普遍适用的解决方案的过程。

上面那句话是参考了西安电子科技大学出版的《java设计模式》中对设计模式的理解。

本文，将通过自己的理解或者参考书中内容。如有不对的地方，你也找不到我。

创建型模式

一般写在前面的话，都是对内容的极大的总结，但是往往在没有后面的实例做铺垫的时候并不能深刻的理解。所以，最好的方式是：多读几遍或者多读几遍。

创建型模式适用于创建对象的模式，抽象了实例化的过程，帮助一个系统独立于其关联对象的创建、组合和表示方式。

功能：

  • 将系统所使用的具体类的信息封装。
  • 隐藏类的实例是如何被创建和组织的。外界对于这些对象只知道他们共同的接口，而不清楚具体实现细节。

作用：

  • 封装创建逻辑。
  • 封装创建逻辑的变化，修改尽量少的代码甚至不修改代码。

话外：功能和作用，有什么区别呢？

功能，前面一边是由名词引导的，比如：这个按钮的功能是什么？

作用，一般是由动词引导的，比如：你按这个按钮有什么用？ 这里的用就是指作用。

当然，你也可以说：这个按钮的作用是指什么？ 这样的逻辑，其实省略了按这个动作。

常见的创建型设计模式

  • 单例模式：一个类，只有一个实例，而且自行实例化，并且向整个系统提供这个实例。
  • 工厂方法模式：工厂类是抽象类，世纪的创建工作又其子类来完成。用意是定义一个创建产品对象的工厂接口，将世纪创建工作推迟到子类中，强调单个对象的变化。
  • 抽象工厂模式：抽象工厂可以向客户提供接口，使客户可以在没有指定产品的具体类型的情况下创建多个产品族中的产品对象，强调系列对象的变化。
  • 建造者模式：将构造对象实例的逻辑移到类的外部，在类的外部定义了该类的构造逻辑。把一个复杂对象的构造过程从对象的表示中分离出来，其结果是将复杂对象简化为一个简单对象，强调产品的构造过程。
  • 原型模式：类似于工厂模式，不同在于原型模式通过复制一个现有对象生成新对象。

结构型模式

是关于类和对象的结构的模式。采用继承机制来组合接口或实现(结结构型模式)，或通过组合一些对象实现新的功能(对象结构性模式)。不同结构性模式的区别在于侧重点不同。

  • 代理模式：为其他对象提供一种代理以控制对该对象的访问。
  • 装饰模式：动态的给一个对象添加一些额外的职责。比生成子类更为灵活。
  • 适配器模式：将一个类的接口编程客户端所需要的接口，从而使原本因为接口不匹配而无法在一起工作的两个类能在一起工作。
  • 组合模式：将对象组合成树形结构以表示部分-整体的层次结构，使得用户对单个对象的组合对象的使用具有一致性。
  • 桥梁模式：将抽象和实现解耦，使其能独立的变化。
  • 外观模式：要求一个子系统的内部和外部的通讯必须同通过一个统一的对象进行，外观模式提供一个高层次的接口。是的子系统更易于使用。
  • 享元模式：是池技术的重要实现方式，使用共享对象可有效地支持大量的细粒度的对象。

行为型模式

关注的是对象的行为,用来解决对象之间的联系问题。

  • 模板方法模式：定义一个操作中的算法的框架，而将一些步骤延迟到子类中，使得子类可以在不改变算法的结构的情况下重写定义该算法某些步骤。
  • 命令模式：是一种高内聚的模式。将一个请求封装成一个对象，从而使用不同的请求把客户端参数化。对请求排队或者记录请求日志，可以提供命令的撤消和恢复功能。
  • 责任链模式：使多个对象都有机会处理请求，从而避免了请求的发送者和接受者之间的耦合关系。
  • 策略模式：定义一组算法，将每个算法都封装起来，并且使他们之间可以互换。
  • 迭代器模式：提供一种方法访问一个容器对象中的各个元素，而又不需要暴露该对象的内部结构。
  • 中介者模式：用一个中介对象封装一系列对象交互，中介者使各个对象不需要显示相互作用，从而使其耦合松散，而且可以独立地改变他们之间的交互。
  • 观察者模式：定义对象间的一对多的依赖关系，使得每当一个对象改变状态，则所有依赖于它的对象都会得到通知并被自动更新。
  • 备忘录模式：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态。
  • 访问者模式：封装一些作用于某种数据结构中的各元素的操作，它可以在不改变数据结构的前提下定义作用于这些元素的新操作。
  • 状态模式：当一个对象内在状态改变时允许其改变行为，这个对象看起来像改变了其类型，状态模式的核心是封装，状态的变更会引起行为的变更。
  • 解释器模式：给定一门语言，定义它的文法的一种表示，并定义一个解释器，该解释器使用该文法表示来解释语言中的句子。

总结

创建型模式适用于创建对象的模式，抽象了实例化的过程，帮助一个系统独立于其关联对象的创建、组合和表示方式。

结构型模式是关于类和对象的结构的模式。采用继承机制来组合接口或实现(结结构型模式)，或通过组合一些对象实现新的功能(对象结构性模式)。不同结构性模式的区别在于侧重点不同。

行为型模式关注的是对象的行为,用来解决对象之间的联系问题。
