设计一个模式的过程就是将问题抽象化，忽略不重要的细节后发现问题本质，并找到普遍适用的解决方案的过程。

上面那句话是参考了西安电子科技大学出版的《java设计模式》中对设计模式的理解。

本文，将通过自己的理解或者参考书中内容。如有不对的地方，你也找不到我。

创建型模式

一般写在前面的话，都是对内容的极大的总结，但是往往在没有后面的实例做铺垫的时候并不能深刻的理解。所以，最好的方式是：多读几遍或者多读几遍。

创建型模式适用于创建对象的模式，抽象了实例化的过程，帮助一个系统独立于其关联对象的创建、组合和表示方式。

功能：

  • 将系统所使用的具体类的信息封装。
  • 隐藏类的实例是如何被创建和组织的。外界对于这些对象只知道他们共同的接口，而不清楚具体实现细节。

作用：

  • 封装创建逻辑。
  • 封装创建逻辑的变化，修改尽量少的代码甚至不修改代码。

话外：功能和作用，有什么区别呢？

功能，前面一边是由名词引导的，比如：这个按钮的功能是什么？

作用，一般是由动词引导的，比如：你按这个按钮有什么用？ 这里的用就是指作用。

当然，你也可以说：这个按钮的作用是指什么？ 这样的逻辑，其实省略了按这个动作。

常见的创建型设计模式

  • 单例模式：一个类，只有一个实例，而且自行实例化，并且向整个系统提供这个实例。
  • 工厂方法模式：工厂类是抽象类，世纪的创建工作又其子类来完成。用意是定义一个创建产品对象的工厂接口，将世纪创建工作推迟到子类中，强调单个对象的变化。
  • 抽象工厂模式：抽象工厂可以向客户提供接口，使客户可以在没有指定产品的具体类型的情况下创建多个产品族中的产品对象，强调系列对象的变化。
  • 建造者模式：将构造对象实例的逻辑移到类的外部，在类的外部定义了该类的构造逻辑。把一个复杂对象的构造过程从对象的表示中分离出来，其结果是将复杂对象简化为一个简单对象，强调产品的构造过程。
  • 原型模式：类似于工厂模式，不同在于原型模式通过复制一个现有对象生成新对象。

结构型模式

是关于类和对象的结构的模式。采用继承机制来组合接口或实现(结结构型模式)，或通过组合一些对象实现新的功能(对象结构性模式)。不同结构性模式的区别在于侧重点不同。

  • 代理模式：为其他对象提供一种代理以控制对该对象的访问。
  • 装饰模式：动态的给一个对象添加一些额外的职责。比生成子类更为灵活。
  • 适配器模式：将一个类的接口编程客户端所需要的接口，从而使原本因为接口不匹配而无法在一起工作的两个类能在一起工作。
  • 桥梁模式：将抽象和实现解耦，使其能独立的变化。
  • 外观模式：要求一个子系统的内部和外部的通讯必须同通过一个统一的对象进行，外观模式提供一个高层次的接口。是的子系统更易于使用。
  • 享元模式：是池技术的重要实现方式，使用共享对象可有效地支持大量的细粒度的对象。

行为型模式

关注的是对象的行为,用来解决对象之间的联系问题。

  • 模板方法模式：定义一个操作中的算法的框架，而将一些步骤延迟到子类中，使得子类可以在不改变算法的结构的情况下重写定义该算法某些步骤。
  • 命令模式：是一种高内聚的模式。将一个请求封装成一个对象，从而使用不同的请求把客户端参数化。对请求排队或者记录请求日志，可以提供命令的撤消和恢复功能。
  • 责任链模式：使多个对象都有机会处理请求，从而避免了请求的发送者和接受者之间的耦合关系。
  • 策略模式：定义一组算法，将每个算法都封装起来，并且使他们之间可以互换。
  • 迭代器模式：提供一种方法访问一个容器对象中的各个元素，而又不需要暴露该对象的内部结构。
  • 中介者模式：用一个中介对象封装一系列对象交互，中介者使各个对象不需要显示相互作用，从而使其耦合松散，而且可以独立地改变他们之间的交互。
  • 观察者模式：定义对象间的一对多的依赖关系，使得每当一个对象改变状态，则所有依赖于它的对象都会得到通知并被自动更新。
  • 备忘录模式：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态。
  • 访问者模式：封装一些作用于某种数据结构中的各元素的操作，它可以在不改变数据结构的前提下定义作用于这些元素的新操作。
  • 状态模式：当一个对象内在状态改变时允许其改变行为，这个对象看起来像改变了其类型，状态模式的核心是封装，状态的变更会引起行为的变更。
  • 解释器模式：给定一门语言，定义它的文法的一种表示，并定义一个解释器，该解释器使用该文法表示来解释语言中的句子。

总结

创建型模式适用于创建对象的模式，抽象了实例化的过程，帮助一个系统独立于其关联对象的创建、组合和表示方式。

结构型模式是关于类和对象的结构的模式。采用继承机制来组合接口或实现(结结构型模式)，或通过组合一些对象实现新的功能(对象结构性模式)。不同结构性模式的区别在于侧重点不同。

二叉树的最大深度

有两种思维方式：一种是自底向上，另一种是自顶向上。

自底向上

先逐渐深入带根节点。然后将根节点的个数+1返回。之所以叫自底向上，是因为我们要深入带最深才能逐渐返回结果。

自顶向下

同样的，就是每往下深入一层，深度（depth）就加1。每一次到不同分支的底层，就更新一次结果。

堆成二叉树

递归起来也不难思考。分别考虑左边的左边是否等于右边的右边和左边的右边是否等于右边的左边。

总结

今天的三个题目就是这样，让我们还回顾一下。

第一题是二叉树的层次遍历，用到了深度优先搜索的思想。
第二题是二叉树的深度的最大值。可以从自顶向上和自底向下两个方向考虑。
第三题是二叉树的对称性，只需要递归的比较left.left=？right.right和left.right=?right.left就可以了。

快速排序

介绍：

快速排序是一种较经典的算法。基本思想就是分治法。即，将一个问题分解成若干个子问题，且子问题和原始问题有同样的规律。又因为有着相同的规律，所以，往往需要用到递归。

而运用到快速排序种，大概就是，将一段数列的排序，分为若干个小排序。我们只需要将每个小排序都排好那整体的排序一定页就排好了。

可以这样思考，一段排好序的数列有什么特点？

通过简单的思考，我们很容易得到这样的结论：

一段排好的数列，取其中任意一个元素X，总有：X左边元素的总是小于X，X右边的总是大于X。

不要小看这个显而易见的结论哦。

那么，我们可以这样，在一段无序的数列中随机取一个数。将比它小的放到它的左边，比他大的放到它右边不就好了吗？

这不还是无序的吗？

那么，我们继续，以刚才那个数为分界线，将左边看成一个数列。执行同样的操作。在左边的数列随机取一个数将比它小的放到它的左边，比他大的放到它右边？一直这样下去。直到剩下3个的时候，我们再随机去一个数。如果正好取到着3个数的中间个那就太好了。将剩下2个中小的放到它左边，大的放到它右边。如果不是正好是中间个，那也没关系，再来一次，剩下2个，随便取那个都可以了。那这最小的子段不久排好了吗？同样的所有段都排好了。

  • TO SAVEPOINT savepoint_name：可选的。 ROLLBACK语句撤消当前会话的所有更改，直到由savepoint_name指定的保存点。 如果省略该子句，则所有更改都将被撤消。
  • FORCE ‘string’：可选的。它用于强制回滚可能已损坏或有问题的事务。 使用此子句，可以将单引号中的事务ID指定为字符串。 可以在系统视图中找到名为DBA_2PC_PENDING的事务标识。
  • 必须拥有DBA权限才能访问系统视图：DBA_2PC_PENDING和V$CORRUPT_XID_LIST。
  • 您无法将有问题的事务回滚到保存点。

git

重要概念；

  • 暂存区

    • 工作区(workspace)
    • 暂存区(stage)
    • 当前版本(master)

指令

初始化：

git init

把文件添加到仓库:三种类型

git add <filename>
git add <filename1> <filename2>
git add .

把文件提交到仓库：message自行修改

git commit -m "message"

查看当前仓库状态：

git status

查看某个修改的内容：

git diff <filename>

查看日志记录：

git log
git log --pretty=oneline

回到前面的版本：

git reset --hard HEAD^
git reset --hard HEAD^^
git reset --hard HEAD~100

查看历史提交的命令(用于查找commit id)：

git reflog

我们使用git add 将修改的文件提交到stage，再使用git commit一次性将stage中的所有提交到master（这里指master这个分支）里。

丢弃工作区的内容：（本质是用版本库替换掉工workspace的内容,也可以看作是切换分支）
若stage中有内容就恢复成stage,没有则恢复master中的。

git checkout -- <filename>

创建并切换分支：

git checkout -b <branch>

将stage中的恢复到workspace中

git reset HEAD <filename>

删除文件

rm <filename>

远程仓库github的设置

创建SSH Key:

ssh-keygen -t rsa -C "meail@mail.com"

再在GitHub里添加用户路径下的 id_rsa.pub公钥。

关联一个远程仓库：

git remote add origin git@server-name:path/repo-name.git

推送master分支到远程仓库：(第一次需要加 -u ，以后不需要)

git push -u origin master

从远程仓库clone 一个仓库：

git clone git@github.com:huixionghexiyi/gitskills.git

删除本地仓库和远程仓库的关联关系：

git remote remove origin

分支

创建分支

git checkout -b dev
或  
git branch dev
git checkout dev

查看分支

git branch

切换分支 

git checkout <branch>

合并分支

将目标分支合并到当前分支
git merge <目标分支>

删除分支

git branch -d <branch>

强制删除

git branch -D <branch>

带参数的查看日志记录

git log --graph --pretty-oneline --abbrev-commit

--graph   表示查看图表
--pretty-oneline 单行查看
--abbrev-commit  缩写的commit id

分支的知识点：

通常情况下：合并分支的时候会默认使用Fast forward 方式。（即文件没有被修改/删除，文件只是被添加（ps:我自己的理解））

禁止Fast forward方式的合并

git merge --no-ff -m "message" <branch>

分支策略：

master应该是非常稳定的，dev是不稳定的，干活的话，就是每个人就从dev上创建自己的分支。

对付bug分支：

stash，储藏(临时保存点)

保存当前工作区的进度

git stash

查看进度的储存list

git stash list

读取进度

git stash apply
git stash drop

或

git stash pop

读取指定的进度

git stash apply stash@{0}

协作

多人需要协作时。在github的项目仓库的设置中的Collaborators选项中将对方邀请进来即可。对方接受，他就对项目可以有写的权限。但是只有项目的拥有者可以选择合并哪个分支到master上（ps：自己的理解）。

标签

创建标签

git tag <标签名>

对历史commit创建标签：

git tag <标签名>  <commit id>

查看标签

git show

创建带说明的标签：

git tag -a <标签名> -m "message" <commit id>

注意，标签总是和commit挂钩。当commit出现在master和dev中那么，这两个分支都可以看见标签。

删除标签：

git tag -d <标签名>

推送标签到远端：

git push origin <tagname>

git push origin --tags

删除远端的标签：

  • 先删除本地的

    git tag -d <标签名>

  • 再删除远端的

    git push origin :refs/tags/<标签名>

忽略文件

忽略文件的原则是：

忽略文件的原则是：

  • 忽略操作系统自动生成的文件，比如缩略图等；
  • 忽略编译生成的中间文件、可执行文件等，也就是如果一个文件是通过另一个文件自动生成的，那自动生成的文件就没必要放进版本库，比如Java编译产生的.class文件；
  • 忽略你自己的带有敏感信息的配置文件，比如存放口令的配置文件。

git config --global alias.st status

    $ cat .gitconfig
[alias]
    co = checkout

总结

每次commit之后都有一个commit id ，而所有的其他操作都是围绕着commit id在运转。所以在项目中commit之前一定要想好。
