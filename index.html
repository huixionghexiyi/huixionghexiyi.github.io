工厂模式

又叫虚拟构造函数(VIrtual Constructor)模式或者多态性工厂模式。工厂方法模式定义一个创建产品对象的工厂接口，将实际创建性工作推出到子类中。

英文原话：

Define an interface for creating an object,but let subclasses decide which class to instantiate.Factory Method lets a class defer instantiation to sub classes.

定义一个用于创建对象的接口，让子类决定实例化哪个类。工厂方法使一个类的实例化延迟到其子类。

  • 简单工厂模式:一个工厂类在产品类实例化的中心位置，它知道每一个产品，它决定哪一个产品类应当被实例化。有点：允许客户端相对独立与产品的创建过程，并且在系统引入新产品的时候无需修复客户端，即在某种程度上支持“开-闭”原则。缺点：支持“开-闭”原则的力度不够，一旦有新的产品加入到系统中，就需要修改工厂类，并将逻辑加入到工厂类中。
  • 工厂方法模式:是对简单工厂模式的进一步抽象。避免了其缺点，保留了优点。核心的工厂类不再负责所有产品的创建，而是将具体的创建工作交给其子类。这个核心类成为一个抽象工厂的角色，仅负责给出具体工厂子类必须实现的接口，不接触细节。
  • 抽象工厂模式:工厂方法模式是针对一个产品等级结构，而抽象工厂模式则需要面对多个产品等级结构。

工厂方法模式

工厂方法模式的四种角色：

  • 抽象工厂角色（Creator）：该角色是工厂方法模式的核心，与应用系统无关，任何创建对象的工厂类都必须实现它.
  • 具体工厂角色（Concrete Creator）：该角色实现了抽象工厂接口，含有与应用密切相关的逻辑，并且通过调用应用程序来创建产品对象。
  • 抽象产品角色（Product）：该角色负责定义产品的共性，实现对产品最抽象的定义。
  • 具体产品角色（Concrete Product）：改角色实现抽象产品角色所声明的接口，工厂方法模式所创建的每一个对象都是某个具体产品角色的实例。

工厂方法模式的应用：

优点：

  • 封装良好，代码结构清晰。创建对象室友约束条件的，如果一个调用者需要使用一个具体的产品对象，只需要知道这个产品对象的类名或约束字符串即可，不用知道创建对象的过程如何。降低了模块间的耦合度。
  • 可扩展：在要增加产品类时，只要修改具体的工厂类或扩展一个工厂类就可以适应变化。
  • 屏蔽产品类：产品类的实现如何变化，调用者不需要关系，只需要关系产品的接口，只要接口不变，上层模块就不需要变化。因为产品的实例化是由工厂负责的，具体生产何种产品对象时由不同的工厂类决定的。
  • 解耦框架：高层模块只需要知道产品的抽象类，其他实现类都不需要关心。符合迪米特法则、依赖倒置原则（只依赖产品类的抽象）、里氏替换原则（可以使用子类替换父类产品）。

使用场景：

  • 工厂方法模式是创建一个方法的替代品，因此在所有需要生成对象的地方都可以使用，但是需要慎重考虑是否需要增加一个工厂类进行管理，增加代码的复杂度。
  • 灵活、可扩展的框架可使用工厂模式。
  • 用在异构项目中。如：通过WebService与一个非java项目交互。
  • 测试驱动开发的框架下使用。如：测试类A就需要同时产生类A关联的类B，使用工厂方法模式可以虚拟出类B，避免耦合。
  • 与其他模式混合使用。从而构造出扩展性更好的设计。

抽象工厂模式

英文原话：

Provide an interface for creating families of related or dependent objects without specifying their concrete classes.

为创建一组相关或相互依赖的对象提供一个接口，而且无需指定他们的具体类。

当有多个业务品种需要分类时，通过抽象工厂模式产生需要的对象时一种非常好多解决方法。

四种角色：

与工厂方法模式相同。

针对的是一族产品。如果产品族中只有一族产品，则等同于工厂方法模式。

优点：

约束为非公开状态，在不同的工厂中，各种产品之间可能具有不同的相互依赖关系，这些依赖关系由工厂封装在其内部对于工厂的使用者来说不可见。

扩展非常容易。若要针对同一产品族简历新的生产线，只需实现产品族中所有产品接口并建立新的工厂类即可。

缺点：

产品族本身的扩展非常困难。如果产品族中增加一个新的产品类型，则需要修改多个接口，并影响已有的工厂类。

应用场景：

当一个对象族（或是一组没有任何关系的对象）都有一组相同的约束时，可以使用抽象工厂模式。

例如：应用需要涉及不同的操作系统时，可以考虑使用抽象工厂模式，如果需要在三个不同的平台（Windows、Linux、Android）上运行，则可以通过抽象工厂模式屏蔽操作系统对应用的影响。三个擦欧总系统的功能、逻辑、UI都应该非常相似，不同的时调用的工厂方法不同。有不同的产品类去处理其与操作系统交互的信息。

单词子集

题目讲得可能让人有点迷糊，简单再介绍一下题目的意思。大概意思是：
在String[] A中有很多单词，String[] B中有很多单词片段(单词片段就是，从一个单词中提取出来的字母的组合，没有顺序)。

要求，找出A中包含B中所有单词片段的单词。注意：是找出单词。

思路：让每个单词中字母的数量和片段B中同一字母在所有片段中出现的次数的最高值做比较。

如果比前者所有的字母的数量都比后者多，那么这个单词满足条件。

具体实现：通过Map是一种比较容易想到的方法。为A中没每个单词申请一个Map，存入字母和它在单词中的数量。将B中所有片段的同一字母和它出现次数的最高值保存到一个Map中。

当然，数组也可以看成一种Map。

和第二天的类似。这里我详细讲一下下标的问题。先看代码：

为什么总是要和中序遍历搭配呢？因为只有中序才能将一个序列分为两部分，利用递归求出二叉树的结构。

在递归方法中，我们通过前序遍历的特点，知道了第一个是head节点，而在中序遍历中这个节点将一个序列分为两部分。我们只要找出在中序遍历中的这两部分序列的起始和结束节点就能将序列划分。

然后根据在中序遍历中结束节点的位置-起始节点的位置=序列长度，然后这个长度再拿到前序遍历中，前序遍历的head节点的位置+1再加上这个距离就是前半部分的结束坐标。后半部分的坐标就是剩下的右边的元素了。

数学上了说就是：

设：前序遍历的序列为pre[pl]~pre[pr]，中序遍历的序列为in[il]~in[ir]

那么，head = pre[pl]

设 in[x] = head (il<=x<=ir)

则，x左边的序列为 left[il]~left[x-1]

则，x右边的序列为 right[x+1]~right[ir]

求得：left的长度 ll = x-1-il

求得：right的长度 rl = ir-x-1

则在前序遍历中的序列就很好求了。就是起始坐标+长度 即可。

单例模式(Singleton)

原话是：

Ensure a class has only one instance,and provide a global point of access to it.

确保一个类只有一个实例，并且向整个系统提供这个实例。

java中实现单例模式通常有两种表现形式：

  • 饿汉模式单例：类加载时，就进行对象的实例化。
  • 懒汉模式单例：第一次引用类时，才进行对象的实例化。

饿汉模式

知识点：

static关键字：static修饰的变量或方法是属于类的，变量在类加载的时候就已经实例化了。static修饰的方法所有的对象使用的是同一个方法。

private关键字：这里Singleton只能生成单个的实例的根本原因是构造方法被private修饰，不能通过对象调用。只能通过static修饰的静态方法调用。

懒汉模式

知识点：

synhronized关键字：同步，确保在多线程的环境下也是单例，因为，当多个线程同时调用这个方法时，由于使用了synhronized修饰，所以产生锁。允许最先到的线程访问，其他线程等待。当锁消除时，再去访问，实例已经被创建了。直接返回即可。

为什么懒汉模式不用sunhronized修饰呢？因为实例已经被创建了。即使再多线程中同时调用也不会产生多例。

单例模式的优点

  • 频繁被创建和销毁的场景里，使用单例模式减少内存开支。
  • 创建一个对象需要较多的资源和性能开支的场景里（读取配置文件、产生其他依赖对象时），使用单例模式减少内存开支。
  • 避免多重占用（如：写文件）。避免了同一资源的同时操作。
  • 可以在系统设置全局访问点，优化和共享资源。（如：设计一个单例类，负责所有数据表的映射处理）

单例模式的缺点

  • 无法创建子类，扩展困难。（扩展只能修改代码）
  • 不利于测试。在并行开发的环境中，若采用单例模式的类没有完成，程序无法测试。
  • 由于单例类通常不实现接口。所以无法使用mock的方式虚拟一个对象。(我也不知道mock是什么方式)。

使用场景

  • 要求生成唯一序列号的环境。
  • 项目需要一个共享访问点或共享数据。（如：web页面中的计数器，不必存放到数据库中使用单例模式能保持数值，并能确保线程安全）
  • 创建一个对象需要消耗的资源过多，如访问IO和数据库等资源。
  • 需要大量静态常量和静态方法（如：工具类）的环境，（当然，也可以直接采用static声明的方式）。

在Spring框架中每个Bean默认的就是单例模式。java基础库中java.lang.Runtime也是单例模式，其中getRuntime()方法返回了唯一实例。

注意事项

单例类分为有状态和无状态:

  • 有状态单例类：有状态的单例类一般是可变的，通常当作状态库使用。如：给系统提供唯一的序列号。
  • 无状态单例类:无状态单例类的对象一般是不变的，通常用来提供工具性的功能方法。如：IO或数据库等。

创建型模式

一般写在前面的话，都是对内容的极大的总结，但是往往在没有后面的实例做铺垫的时候并不能深刻的理解。所以，最好的方式是：多读几遍或者多读几遍。

创建型模式适用于创建对象的模式，抽象了实例化的过程，帮助一个系统独立于其关联对象的创建、组合和表示方式。

功能：

  • 将系统所使用的具体类的信息封装。
  • 隐藏类的实例是如何被创建和组织的。外界对于这些对象只知道他们共同的接口，而不清楚具体实现细节。

作用：

  • 封装创建逻辑。
  • 封装创建逻辑的变化，修改尽量少的代码甚至不修改代码。

话外：功能和作用，有什么区别呢？

功能，前面一边是由名词引导的，比如：这个按钮的功能是什么？

作用，一般是由动词引导的，比如：你按这个按钮有什么用？ 这里的用就是指作用。

当然，你也可以说：这个按钮的作用是指什么？ 这样的逻辑，其实省略了按这个动作。

常见的创建型设计模式

  • 单例模式：一个类，只有一个实例，而且自行实例化，并且向整个系统提供这个实例。
  • 工厂方法模式：工厂类是抽象类，世纪的创建工作又其子类来完成。用意是定义一个创建产品对象的工厂接口，将世纪创建工作推迟到子类中，强调单个对象的变化。
  • 抽象工厂模式：抽象工厂可以向客户提供接口，使客户可以在没有指定产品的具体类型的情况下创建多个产品族中的产品对象，强调系列对象的变化。
  • 建造者模式：将构造对象实例的逻辑移到类的外部，在类的外部定义了该类的构造逻辑。把一个复杂对象的构造过程从对象的表示中分离出来，其结果是将复杂对象简化为一个简单对象，强调产品的构造过程。
  • 原型模式：类似于工厂模式，不同在于原型模式通过复制一个现有对象生成新对象。

结构型模式

是关于类和对象的结构的模式。采用继承机制来组合接口或实现(结结构型模式)，或通过组合一些对象实现新的功能(对象结构性模式)。不同结构性模式的区别在于侧重点不同。

  • 代理模式：为其他对象提供一种代理以控制对该对象的访问。
  • 装饰模式：动态的给一个对象添加一些额外的职责。比生成子类更为灵活。
  • 适配器模式：将一个类的接口编程客户端所需要的接口，从而使原本因为接口不匹配而无法在一起工作的两个类能在一起工作。
  • 组合模式：将对象组合成树形结构以表示部分-整体的层次结构，使得用户对单个对象的组合对象的使用具有一致性。
  • 桥梁模式：将抽象和实现解耦，使其能独立的变化。
  • 外观模式：要求一个子系统的内部和外部的通讯必须同通过一个统一的对象进行，外观模式提供一个高层次的接口。是的子系统更易于使用。
  • 享元模式：是池技术的重要实现方式，使用共享对象可有效地支持大量的细粒度的对象。

行为型模式

关注的是对象的行为,用来解决对象之间的联系问题。

  • 模板方法模式：定义一个操作中的算法的框架，而将一些步骤延迟到子类中，使得子类可以在不改变算法的结构的情况下重写定义该算法某些步骤。
  • 命令模式：是一种高内聚的模式。将一个请求封装成一个对象，从而使用不同的请求把客户端参数化。对请求排队或者记录请求日志，可以提供命令的撤消和恢复功能。
  • 责任链模式：使多个对象都有机会处理请求，从而避免了请求的发送者和接受者之间的耦合关系。
  • 策略模式：定义一组算法，将每个算法都封装起来，并且使他们之间可以互换。
  • 迭代器模式：提供一种方法访问一个容器对象中的各个元素，而又不需要暴露该对象的内部结构。
  • 中介者模式：用一个中介对象封装一系列对象交互，中介者使各个对象不需要显示相互作用，从而使其耦合松散，而且可以独立地改变他们之间的交互。
  • 观察者模式：定义对象间的一对多的依赖关系，使得每当一个对象改变状态，则所有依赖于它的对象都会得到通知并被自动更新。
  • 备忘录模式：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态。
  • 访问者模式：封装一些作用于某种数据结构中的各元素的操作，它可以在不改变数据结构的前提下定义作用于这些元素的新操作。
  • 状态模式：当一个对象内在状态改变时允许其改变行为，这个对象看起来像改变了其类型，状态模式的核心是封装，状态的变更会引起行为的变更。
  • 解释器模式：给定一门语言，定义它的文法的一种表示，并定义一个解释器，该解释器使用该文法表示来解释语言中的句子。

总结

创建型模式适用于创建对象的模式，抽象了实例化的过程，帮助一个系统独立于其关联对象的创建、组合和表示方式。

结构型模式是关于类和对象的结构的模式。采用继承机制来组合接口或实现(结结构型模式)，或通过组合一些对象实现新的功能(对象结构性模式)。不同结构性模式的区别在于侧重点不同。

行为型模式关注的是对象的行为,用来解决对象之间的联系问题。

路径总和

还记得上次前面的文章讲的自顶向下和自底向上吗？考虑一下两种思路都可以吗？

当遇到树问题时，请先思考一下两个问题：

  • 你能确定一些参数，从该节点自身解决出发寻找答案吗？
  • 你可以使用这些参数和节点本身的值来决定什么应该是传递给它子节点的参数吗？

如果答案都是肯定的，那么请尝试使用 “自顶向下” 的递归来解决此问题。

或者你可以这样思考：

  • 对于树中的任意一个节点，如果你知道它子节点的答案，你能计算出该节点的答案吗？

如果答案是肯定的，那么 “自底向上” 的递归可能是一个不错的解决方法。

我们能确定一些参数传递给子节点吗？是的，可以，我们将sum减去自身传递下去，知道结束条件，并且等于0的时候。
